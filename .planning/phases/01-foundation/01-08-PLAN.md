<objective>
Fix two UAT-confirmed gaps that prevent the frontend from being usable:

1. **Tailwind CSS utility classes not generated** (Tests 3 & 4 — unstyled pages): Tailwind v4 with Next.js + Turbopack requires a `postcss.config.mjs` file that registers `@tailwindcss/postcss` as a PostCSS plugin. With `mode: 'onBlur'`, `isValid` starts `false` and only updates after each field is explicitly blurred. The button `disabled={!isValid}` is therefore permanently disabled unless the user manually tabs away from every field — it visually appears to do nothing.

Purpose: These two fixes unblock all remaining UAT tests (4–16) that were skipped or failed due to missing styles and broken registration.

<task type="auto">
  <name>Task 1: Create postcss.config.mjs to activate Tailwind v4 utility class generation</name>
  <files>frontend/postcss.config.mjs</files>
  <action>
**Root cause:** Tailwind CSS v4 (`^4.0.0`) ships `@tailwindcss/postcss` as a separate package (already in `package.json`). Without a `postcss.config.mjs`, Next.js + Turbopack never invokes this plugin, so `@import "tailwindcss"` in `globals.css` is treated as a plain CSS import with no utility class generation. Only custom `.glass` and `.btn-gradient` classes (hand-written CSS in globals.css) are emitted. All Tailwind utility classes (`flex`, `grid`, `bg-*`, `text-*`, `rounded-*`, `px-*`, etc.) are absent from the compiled output.

**Fix:** Create `frontend/postcss.config.mjs` with the standard Tailwind v4 PostCSS plugin:

```javascript
// frontend/postcss.config.mjs
export default {
  plugins: {
    '@tailwindcss/postcss': {},
  },
}
```

That is the entire file. No other configuration is needed — Tailwind v4 uses `@import "tailwindcss"` in `globals.css` as the configuration entry point (CSS-first configuration). The `@tailwindcss/postcss` plugin auto-scans all source files for class names.

**Do NOT:**
- Add a `tailwind.config.js` or `tailwind.config.ts` — not needed for Tailwind v4
- Add `postcss-preset-env` or `autoprefixer` — not needed; Tailwind v4 handles vendor prefixes internally
- Change `globals.css` — `@import "tailwindcss"` is already the correct Tailwind v4 CSS entry point

<task type="auto">
  <name>Task 2: Fix SignupForm validation mode so Create Account button enables on live input</name>
  <files>frontend/src/components/auth/SignupForm.tsx</files>
  <action>
**Root cause:** `SignupForm` uses `mode: 'onBlur'` in `useForm`. With this mode, `isValid` only updates after each field fires a blur event. The button `disabled={!isValid || registerMutation.isPending}` starts disabled (all fields empty → isValid=false) and only becomes enabled once EVERY field has been blurred AND passes validation. Users who fill fields without blurring each one (e.g., clicking "Create Account" directly) find the button permanently disabled — it "does nothing".

**Fix:** Change `mode: 'onBlur'` to `mode: 'onChange'` — exactly as `LoginForm` does (already working). This makes `isValid` re-evaluate on every keystroke so the button enables as soon as all fields pass validation.

In `frontend/src/components/auth/SignupForm.tsx`, find the `useForm` call:

```typescript
// BEFORE (broken):
const { register, handleSubmit, watch, formState: { errors, isValid } } = useForm<SignupFormData>({
  resolver: zodResolver(signupSchema),
  mode: 'onBlur',
})

// AFTER (fixed):
const { register, handleSubmit, watch, formState: { errors, isValid } } = useForm<SignupFormData>({
  resolver: zodResolver(signupSchema),
  mode: 'onChange',
})
```

**Only change** `'onBlur'` → `'onChange'`. Do not modify any other part of the file.

**Why `onChange` is correct:**
- `LoginForm` already uses `mode: 'onChange'` (working correctly per UAT)
- With `onChange`, `isValid` updates on every keystroke — button enables immediately when all fields pass Zod validation
- Error messages still appear (they use `errors` from `formState`, which is populated on first invalid change)
  </action>
  <verify>

  </verify>
  <done>

- `grep "mode: 'onBlur'" frontend/src/components/auth/SignupForm.tsx` returns no match
- TypeScript strict mode passes with no errors
- The Create Account button enables as soon as all fields pass validation (no blur required)
  </done>
