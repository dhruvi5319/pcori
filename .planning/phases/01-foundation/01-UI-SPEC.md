## Spacing Scale

Declared values (all multiples of 4 — 8-point grid):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline badge padding, tight list item gaps |
| sm | 8px | Input internal padding (vertical), compact element gaps, icon-to-label spacing |
| md | 16px | Default element spacing, form field gaps, card internal padding |
| lg | 24px | Section padding within cards, dialog internal padding |
| xl | 32px | Layout column gaps, section breaks within a page |
| 2xl | 48px | Major section breaks (hero → features grid), card grid row gap |
| 3xl | 64px | Page-level vertical spacing (hero section top/bottom) |

**Exceptions:**
- Touch targets (interactive buttons, nav items): minimum 44px height — exceeds grid to meet WCAG 2.1 AA touch target requirement
- Sidebar collapsed width: 56px (icon-only; centered 24px icon + 16px padding each side)
- Sidebar expanded width: 240px
- Auth card max-width: 400px (centered)
- Landing hero section min-height: 600px

## Color

### Semantic Token Contract

All color values are expressed as Tailwind CSS 4 CSS custom properties. The system is adaptive (light/dark via next-themes) — both light and dark values must be declared.

| Role | Light Mode Value | Dark Mode Value | Usage |
|------|-----------------|-----------------|-------|
| **Dominant (60%)** | `#FFFFFF` (white) | `#0A0A0A` (near-black) | Page background, auth page background, form backgrounds |
| **Secondary (30%)** | `#F4F6F9` (cool gray-50) | `#141414` (dark surface) | Auth card surface, sidebar background (glassmorphism applied on top), header background (glassmorphism applied on top), input field backgrounds |
| **Accent — Brand Blue (10%)** | `#1D4ED8` (blue-700) | `#3B82F6` (blue-500) | Reserved for: primary CTA buttons ("Sign In", "Create Account", "Reset Password", "Get Started"), active sidebar nav item indicator, focus ring color |
| **Destructive** | `#DC2626` (red-600) | `#EF4444` (red-500) | Destructive confirmation buttons only (no Phase 1 destructive actions in auth screens; defined here for shell completeness) |

**Accent reserved for (explicit list):**
1. Primary CTA buttons (Sign In, Create Account, Send Reset Link, Reset Password, Get Started)
2. Active/selected sidebar navigation item left-border indicator and icon fill
3. Focus rings on all interactive elements (keyboard navigation)
4. Landing page "Get Started" hero button gradient base color

> Accent is NOT used for: links, secondary buttons, nav item hover states, error states, or status badges. Those use their own semantic colors.

### Status Color Reference (App Shell — for Phase 2+ screens; define tokens in Phase 1)

| Status | Color | Text Label Required |
|--------|-------|---------------------|
| `CLASSIFIED` | green (`#16A34A` / `#22C55E` dark) | "Classified" |
| `PROCESSING` | blue pulsing (`#2563EB` / `#3B82F6` dark) | "Processing" |
| `PENDING` | gray (`#6B7280` / `#9CA3AF` dark) | "Pending" |
| `FAILED` | red (`#DC2626` / `#EF4444` dark) | "Failed" |
| `NEEDS_REVIEW` | amber (`#D97706` / `#F59E0B` dark) | "Needs Review" |

> WCAG mandate: Color is NEVER the sole status indicator. Every status badge must include its text label alongside the color dot/indicator.

### Advanced Visual Effects (User Decision #5 — Locked)

The following effects are required for Phase 1 and must be implemented precisely:

#### Glassmorphism (Sidebar + Header)
- `backdrop-filter: blur(12px)` (or `backdrop-blur-xl` in Tailwind)
- Background: `rgba(255,255,255,0.6)` light / `rgba(10,10,10,0.6)` dark
- Border: `1px solid rgba(255,255,255,0.15)` light / `1px solid rgba(255,255,255,0.08)` dark
- Applied to: sidebar panel, app header (sticky), auth card on landing page

#### Landing Page Hero Section
- Background: dark section with CSS gradient mesh (`background: radial-gradient(ellipse at 20% 50%, #1e3a5f 0%, #0f172a 40%, #1a1040 100%)`)
- Text: white/near-white against dark hero
- Feature cards: gradient border on hover (`border: 1px solid transparent; background-clip: padding-box` with `::before` gradient pseudo-element — blue-to-purple)
- Animated counters on KPI elements (landing page marketing stats, if any)

#### CTA Gradient Accents
- Primary CTA buttons use gradient fill: `linear-gradient(135deg, #1D4ED8 0%, #7C3AED 100%)` (blue-to-purple)
- Applied to: "Get Started" hero button, "Sign In" button, "Create Account" button
- Hover state: gradient shifts slightly brighter + `transform: translateY(-1px)`

#### Auth Card Elevation
- Auth card (Login, Signup, Forgot Password, Reset Password): soft shadow `box-shadow: 0 8px 32px rgba(0,0,0,0.12)` light / `0 8px 32px rgba(0,0,0,0.4)` dark
- Card hover does NOT trigger lift (auth cards are not interactive cards)

## Screen-by-Screen Visual Contract

### Screen 1: Landing Page (`/`)

**Layout zones:**
1. **Public Nav bar** — Logo left, "Login" + "Sign Up" buttons right. Height 64px. No glassmorphism (clean white/dark bar).
2. **Hero Section** — Full-width dark gradient mesh background. Min-height 600px. Centered content column (max-width 720px).
   - Headline (Display 48px / 600): "Automate Research Plan Classification"
   - Subheadline (Body 16px / 400): "Upload a PDF. Get AI-powered PCORI taxonomy classification in minutes."
   - Dual CTAs: "Get Started →" (gradient accent button, primary) + "Sign In" (ghost/outline button, secondary)
   - Animated counter KPIs below hero text (e.g., "143 Plans Classified" with count-up animation on first scroll into view)
3. **Features Grid** — 3 cards in a row (responsive: 1 col on mobile). White/dark-surface background (secondary color).
   - Card: icon (lucide) + title (Heading 24px / 600) + description (Body 16px / 400)
   - Gradient border on card hover (blue-to-purple gradient border, 1px)
   - Feature 1: "AI Classification" — icon: `Brain` — "Upload PDF → classified in minutes"
   - Feature 2: "Full Audit Trail" — icon: `ClipboardCheck` — "Every decision logged with reviewer name and timestamp"
   - Feature 3: "Analytics & Reporting" — icon: `BarChart3` — "Real-time KPIs and one-click Excel export"
4. **Footer** — `© PCORI Research Analytics Platform · Privacy · Terms` — Label 14px / 400, muted color

### Screen 2: Login (`/login`)

**Layout:** Centered AuthCard (max-width 400px) vertically and horizontally centered on page. Public layout (no sidebar). Light/dark adaptive background.

**Auth Card contents:**
- Card title: "Sign In to PCORI Analytics" (Heading 24px / 600)
- Subtext: "Enter your credentials to continue" (Label 14px / 400, muted)
- Username field: label "Username", text input, required
- Password field: label "Password", PasswordInput component (show/hide toggle), required
- "Sign In" button: full-width, gradient accent, disabled until both fields non-empty
- Below button (Label 14px, right-aligned link): "Forgot password?"
- Below that (centered): "Don't have an account? [Sign up]"

**Interaction states (all must be implemented):**

| State | Visual | Copy |
|-------|--------|------|
| Default | Fields empty; Sign In disabled (opacity 50%) | — |
| Fields filled | Sign In enabled (full opacity) | — |
| Loading | Sign In: spinner replacing icon + disabled | "Signing in…" |
| Error: invalid credentials | sonner toast (red) | "Invalid username or password" |
| Error: account locked | sonner toast (red) | "Account locked. Try again after [X] minutes." |
| Error: email not verified | sonner toast (amber) | "Verify your email first." + inline "Resend verification email" link |
| Error: account inactive | sonner toast (red) | "This account has been deactivated. Contact your administrator." |
| Success | Redirect to `/dashboard` | — |

### Screen 3: Sign Up (`/signup`)

**Layout:** Centered AuthCard (max-width 400px). Public layout.

**Auth Card contents:**
- Card title: "Create your account" (Heading 24px / 600)
- Fields (in order): Username *, Email *, First Name *, Last Name * (side-by-side on ≥400px), Password *
- Username helper: "3–50 characters, letters, numbers, and underscores only" (Label 14px, muted)
- Password rules: inline checklist (uppercase ✓, lowercase ✓, digit ✓) — shows on focus
- "Create Account" button: full-width, gradient accent, disabled until all fields pass zod validation
- Validation: zod on blur per field; submit button disabled until valid
- Below button: "Already have an account? [Sign in]"

**Interaction states:**

| State | Visual | Copy |
|-------|--------|------|
| Default | All fields empty; button disabled | — |
| Per-field validation (blur) | Red inline error below field | e.g. "Username already in use", "Email already registered", "Password must contain an uppercase letter" |
| Loading | Button: spinner + disabled | "Creating account…" |
| Success | sonner toast (green) → redirect `/login` | "Account created — check your email to verify" |

### Screen 4: Forgot Password (`/forgot-password`)

**Layout:** Centered AuthCard (max-width 400px). Public layout.

**Auth Card contents:**

- Subtext: "Enter your email and we'll send you a reset link." (Label 14px, muted)
- Email field: label "Email address", text input

**Interaction states:**

| State | Visual | Copy |
|-------|--------|------|

| Loading | Button: spinner + disabled | "Sending…" |

### Screen 5: Reset Password (`/reset-password?token=...`)

**Layout:** Centered AuthCard (max-width 400px). Public layout.

**Auth Card contents:**

- New Password field (PasswordInput with show/hide)

- Password rules checklist (inline — same as Signup)
- "Reset Password" button: full-width, gradient accent, disabled until both fields pass validation and passwords match
- Validation: passwords must match (client-side, zod `refine`)

**Interaction states:**

| State | Visual | Copy |
|-------|--------|------|
| Default | Fields empty; button disabled | — |
| Passwords mismatch | Inline error on Confirm field | "Passwords do not match" |

| Loading | Button: spinner + disabled | "Updating password…" |
| Success | sonner toast (green) → redirect `/login` | "Password updated. You can now sign in." |

### Screen 6: Email Verification (`/verify-email?token=...`)

**Layout:** Centered AuthCard (max-width 400px). Public layout.

**States:**

| State | Visual | Copy |
|-------|--------|------|

| Success | CheckCircle icon (green, 48px) | "Email verified!" + "[Log In Now]" button (→ `/login`) |
| Expired/invalid token | XCircle icon (red, 48px) | "This verification link has expired or was already used." + "[Request new verification email]" button |

## Interaction & Animation Contract

### Global Interaction Patterns (all Phase 1 screens)

| Pattern | Specification |
|---------|---------------|
| **Loading — buttons** | Spinner (lucide `Loader2` with `animate-spin`) replaces button text/icon; button disabled |
| **Loading — page navigation** | No full-page spinner; instant navigation with Next.js App Router |
| **Form validation** | zod validation fires on `blur` per field; form submit blocked until all fields pass |
| **Submit disabled state** | Button opacity 50% + `cursor-not-allowed` (Tailwind `disabled:opacity-50 disabled:cursor-not-allowed`) |
| **Focus ring** | Accent color ring: `focus-visible:ring-2 focus-visible:ring-blue-600 focus-visible:ring-offset-2` |
| **Toast position** | Top-right, rich colors, max 3 visible at once (sonner defaults) |
| **Redirect after auth** | Next.js `router.push()` — never `window.location` |
| **Session expiry** | TanStack Query 401 interceptor → clear tokens → redirect `/login` → amber toast |

### Advanced Animations (Phase 1 specific)

| Element | Animation | Implementation |
|---------|-----------|----------------|
| Landing hero animated counters | Count-up from 0 to target value on first scroll into view | `useIntersectionObserver` hook + `requestAnimationFrame` counter |
| Feature card gradient border | CSS gradient border appears on hover (0.2s ease) | `::before` pseudo-element with gradient + opacity transition |
| Primary CTA hover | `translateY(-2px)` + brightness(1.1) (0.15s ease) | Tailwind `hover:-translate-y-0.5 hover:brightness-110 transition-all duration-150` |
| Sidebar collapse/expand | Width transition 240px ↔ 56px (0.2s ease-out) | CSS transition on sidebar width; labels fade out with opacity transition |
| PROCESSING badge pulse | Blue pulsing ring around status dot | CSS `animate-ping` (Tailwind built-in); defined in Phase 1 for Phase 2 use |

## Accessibility Contract

All requirements are WCAG 2.1 AA:

| Requirement | Implementation |
|-------------|---------------|
| Color + text always together | Status badges: dot color + text label. Color alone is never an indicator. |
| Contrast — body text | 4.5:1 minimum against background (verified against dominant + secondary color tokens) |
| Contrast — large text (Display 48px) | 3:1 minimum |
| Contrast — UI components | 3:1 for interactive elements (buttons, inputs) |
| Keyboard navigation | All interactive elements reachable via Tab; Radix primitives handle arrow-key nav |
| Focus visible | Explicit focus-visible ring (accent color) on all interactive elements; never `outline: none` |
| Form labels | All form inputs have explicit `<label htmlFor>` linked to input `id` |
| Error associations | `aria-describedby` linking input to its error message element |
| Dialog focus trap | Radix Dialog primitive handles focus trap + Escape-to-close automatically |
| Tooltip accessibility | Radix Tooltip primitive; `aria-label` on icon-only sidebar items (collapsed state) |
| Header icon-only buttons | `ThemeToggle` → `aria-label="Toggle theme"`; `NotificationBell` → `aria-label="View notifications"`; Mobile sidebar trigger → `aria-label="Open navigation"` |
| Touch targets | Minimum 44px height/width for all interactive elements |
| Screen reader | sonner toasts have `role="status"` / `aria-live` (built-in) |
| Password show/hide | Toggle button has `aria-label="Show password"` / `aria-label="Hide password"` |
